# 台股盤後分析工具 — 設計文件

> 本文件是 2026-07-06 需求訪談的結論，為本專案的設計事實來源。
> 實作與後續決策若與本文件衝突，須先更新本文件再動工。

## 1. 定位

- **台股**盤後分析的**自用工具**：查看股價與 K 線、籌碼篩選、技術指標、券商分點追蹤與篩選、簡單策略分析。
- 單一使用者起步，但架構保留產品化的路（見 §2 架構原則）。
- **只用盤後資料**：籌碼與分點本質上就是盤後公布；不做盤中即時報價、不做串流。整個系統是「每天傍晚跑一次 batch pipeline」。

## 2. 架構原則（產品化保留）

第一天就遵守、成本低的原則：

1. **資料層與應用層分離** — 爬蟲/資料擷取寫成獨立 ingestion 模組，每個資料源、通知管道都是 adapter，之後換付費資料源（Shioaji、富果、凱衛）或加通知管道只換 adapter。
2. **不把「單一使用者」寫死** — 自選股、策略設定等從第一天掛在 user 概念下，即使目前只有一筆 user。
3. **Thick server / thin client** — 篩選引擎、指標計算、策略邏輯全在後端 API，前端只負責呈現。未來手機 app 只是同一套 API 的另一個 client。

**先不做**：多租戶、權限系統、計費。等真的產品化再說。

## 3. 資料來源

| 資料 | 來源 | 備註 |
|---|---|---|
| 日價量（K 線） | 證交所/櫃買官方 OpenAPI 與每日收盤行情 | 歷史可整批回補十年以上 |
| 籌碼（三大法人、融資券、借券） | 證交所/櫃買每日統計 | 穩定好爬 |
| **券商分點**（買賣日報表） | 自爬 `bsr.twse.com.tw`（上市）＋櫃買對應頁 | 見下方分點注意事項 |
| 除權息、股本、基本資料 | 公開資訊觀測站 | 格式雜但好爬 |

### 分點資料的關鍵事實

- 官方頁面**只提供最近交易日**，不能回查歷史 → 分點資料庫**從 pipeline 上線日開始累積**，越早上線越好。
- **錯過一天就是永久缺洞**（免費範圍內）。抓取窗口約為當天傍晚公布後至下一交易日資料蓋掉前。
- 一次只能查一檔、每檔可能多頁、頁面帶簡單驗證碼（社群有成熟 OCR 方案）→ 全案爬蟲維運成本最高的一塊，需節流與改版監控。
- **FinMind 贊助方案**（每月數百元台幣等級）留作日後選配：一次回補多年分點歷史、或補自爬缺洞。adapter 介面（如 `BrokerBranchSource`）要讓這個替換不動下游。

## 4. 技術棧

- **後端**：Python + FastAPI。理由：指標計算（pandas-ta / TA-Lib）、回測（vectorbt）、爬蟲參考實作都在 Python 生態。
- **前端**：React + TypeScript 的本機 web app。圖表用 lightweight-charts / ECharts。
- **型別契約**：FastAPI 自動生成 OpenAPI schema → 前端 codegen TypeScript client，避免前後端契約漂移。
- **手機**：先靠 RWD + PWA（加到主畫面）。React Native 延後到確定需要推播/Widget 等原生能力再開——屆時因 thick server 原則，只需寫 UI。
- **資料庫**：PostgreSQL 單庫全包（操作型＋分析型資料）。分點表按日期**按月分區**＋（股票, 分點, 日期）索引。日後若有重度全市場掃描需求，再鏡像 Parquet 給 DuckDB 讀（加法，非重構）。

## 5. 部署與排程

- 跑在**家中常開的 Windows 機**：WSL2 + **Docker Compose**（Postgres、FastAPI、ingestion 全進 compose，`restart: unless-stopped`，重開機自動復活）。
- 開發在 Mac、部署到 Windows，同一份 compose，環境一致。
- 排程用 Python 內建 scheduler（APScheduler），**不依賴 Windows 工作排程器**。
- **每日傍晚起每小時重試直到成功**（不是一天只試一次），對應分點資料錯過即缺洞的特性。
- ingestion 失敗當天推 Telegram 告警（見 §8）。

## 6. 功能與分期

### 第一期（順序即優先序）

1. **Ingestion pipeline** — 最急，分點累積從上線日起算。
2. **K 線與技術指標頁** — 指標用 pandas-ta 標準集。
3. **篩選引擎** — 條件存成結構化 **JSON 條件 schema**，UI 條件建構器組條件（指標比較、連續 N 天、排名/佔比、分點行為）。此 JSON 是引擎的中間表示：篩選頁即時查詢、訊號掃描每日跑、回測期翻譯成向量運算，三者共用。
4. **分點功能** — 個股視角（哪些分點在買賣、集中度、主力成本區）＋分點視角（追蹤特定分點每日進出）＋**手動標籤**（隔日沖、地緣、外資通路…，一張標籤表，內容手動維護）。
5. **存策略＋每日自動掃描＋命中歷史** — 篩選條件存成策略，盤後自動掃全市場，命中結果入庫。命中歷史即免費的前向測試紀錄（無前視偏差）。
6. **Telegram 推播** — 每日訊號＋pipeline 故障告警。

### 第二期（第一期穩定即可動工，不等資料）

- **回測框架**（vectorbt）— 上線即可回測指標類策略（價量歷史第一天就回補完）；**分點類策略約 3~6 個月後**資料夠深再納入。
- 分點**異常掃描**（某分點突然大買平常不碰的股票、單一分點買超佔成交量比例過高等），等用出手感再定義「異常」。
- 策略定義的 **code 逃生口**（受控執行 Python 片段），確定 UI 條件表達不了再加。

### 明確不做（目前）

- 盤中即時報價、串流、tick 資料。
- 自動分點分類（標籤先手動）。
- 多租戶／帳號系統／計費。

## 7. 價格處理：雙價格制

- 資料庫同時存**原始價**與**除權息還原價**（除權息事件抓公開資訊觀測站，還原係數自算）。
- **K 線顯示**：預設原始價（與一般看盤軟體一致），一鍵切換還原線。
- **技術指標計算、篩選條件、回測**：一律**還原價**（避免除息缺口造成假訊號；台股六到九月除息旺季不還原會假訊號淹沒）。
- **分點成本估算、當日報價**：一律**原始價**（分點用真實成交價買）。

## 8. 通知

- **Telegram bot**（免費、無則數限制）：每日訊號命中清單＋ingestion 故障告警。
- 通知做成 adapter，日後可加 LINE Messaging API / email 等。
- 注意：LINE Notify 已於 2025-03 終止服務，勿再參考。

## 9. 設計原則：新手友善的內建說明

凡有隱藏規則或易誤讀處，UI 放輕量提示（ⓘ tooltip 或一行小字），至少包括：

- 原始價／還原價切換旁：一句話差異說明＋「指標與回測用還原價算」。
- 技術指標參數旁的簡短定義。
- 籌碼欄位口語化說明（融資使用率、借券賣出餘額等）。
- 分點頁標注：「分點資料為盤後公布、本系統自 YYYY-MM 起累積」。
- 回測結果頁標注前提假設（手續費、滑價設定）。

此原則兼顧自用學習與未來產品化賣點。

## 10. 已知風險

- **分點爬蟲**是維運熱點：驗證碼失效、官方改版、節流被擋。緩解：故障當日 Telegram 告警＋每小時重試＋FinMind 付費回補作最後手段。
- Windows 機停機一整天以上 → 該日分點永久缺洞（免費範圍內）。對「追蹤近期分點動向」傷害有限；嚴謹回測前可用 FinMind 補洞。
- 官方資料源對高頻請求或海外 IP 可能不友善；本案在家用 IP、盤後 batch，風險低但爬蟲要節流。
